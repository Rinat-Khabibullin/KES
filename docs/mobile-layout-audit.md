# Mobile Layout Audit

Дата: 2026-07-19.

## Проверенные размеры

Проверены production build и `vite preview` на:

- `320x568`
- `360x800`
- `375x812`
- `390x844`
- `412x915`
- `430x932`
- `768x1024`
- `1440x900`

Скриншоты проверки сохранены в `/tmp/kes-mobile-layout-screens`.

## Найденные проблемы

- `@media (max-width: 1120px)` одним общим правилом переводил в одну колонку разные по структуре секции: Hero, районы, гарантии, FAQ, контакты, партнёров и оценку по фото. Из-за этого tablet-композиция ломалась раньше, чем реально требовалось по содержимому.
- В `PhotoEstimate` полезные инструкции, примеры фото и CTA находились в отдельной панели после текстового блока. На mobile это превращалось в длинную колонку, где текст, декоративный телефон, шаги и действия воспринимались как разрозненные блоки.
- Декоративный `photo-estimate__phone` имел мобильную высоту около `238px` и стоял между описанием и инструкцией.
- Главным CTA в блоке фото визуально был звонок, хотя сценарий секции должен вести к отправке фотографии.
- Якорь `#photo-estimate` был на заголовке, а не на секции. При переходе с нижней панели и с учётом sticky header позиция была менее предсказуемой.
- `scrollIntoView` с глобальным `scroll-behavior: smooth` долго прокручивал длинные мобильные переходы и мог оставлять секцию далеко от начала в первые секунды.
- На small mobile заголовок фото-секции занимал слишком много высоты.

## Причины

- Breakpoint был задан по ширине экрана, а не по реальной устойчивости конкретной сетки.
- DOM-порядок секции «Оценка по фото» повторял desktop-композицию, а mobile-порядок пытался получиться только CSS-стеком.
- Декоративный элемент имел собственную карточку и минимальную высоту, поэтому разрывал инструкцию.
- Для hash navigation использовался `scrollIntoView`, зависящий от нативного hash-scroll и CSS `scroll-behavior`.

## Исправленные секции

- `PhotoEstimate`: полностью перестроен mobile-сценарий.
- `Areas`, `Guarantee`, `FAQ`, `Partners`, `Contact`, `Footer`: получили отдельную tablet-адаптацию на `900px` вместо общего правила на `1120px`.
- `Hero`: оставлен в отдельном breakpoint, без смешивания с другими секциями.
- `MobileActions`: скрывается не только при открытом чате, но и при открытом мобильном меню.
- Hash navigation: `#photo-estimate` и другие якоря прокручиваются координатно с учётом sticky header.

## Новый порядок «Оценка по фото»

В DOM и визуально:

1. Бейдж `Оценка по фото`.
2. Заголовок.
3. Короткое объяснение пользы.
4. Chips с примерами: `щит`, `розетка`, `место установки`, `кабельная трасса`, `повреждение`.
5. Упорядоченный список из 3 шагов.
6. Primary CTA `Отправить фото через Авито`.
7. Secondary CTA `Позвонить и описать задачу`.
8. Декоративная телефонная иллюстрация только после полезного сценария и только на tablet/desktop.

## Решения

- `section` фото теперь имеет `id="photo-estimate"` и `aria-labelledby="photo-estimate-title"`.
- Полезные шаги сделаны через `ol`, без CSS-перестановки DOM.
- `photo-estimate__panel` оставлен `aria-hidden="true"` и скрывается на `max-width: 760px`.
- На mobile весь блок фото выглядит как единая карточка с общими отступами, радиусом и фоном.
- Primary CTA в секции фото теперь ведёт на Авито, звонок стал secondary.
- Chips переведены в компактную двухколоночную сетку на mobile.
- На `max-width: 460px` заголовок фото-секции уменьшен локально.
- Общий список секций в `@media (max-width: 1120px)` заменён независимыми правилами. Tablet-перестройка для смысловых двухколоночных секций сделана на `900px`.
- Hash-scroll теперь считает координату цели вручную: `targetTop - stickyHeader - 18px`. На mobile/tablet переход мгновенный, на desktop плавный, но по той же координате.

- `375x812`: overflow нет, фото-сценарий корректен, якорь корректен, mobile CTA и чат корректны.
- `390x844`: overflow нет, фото-сценарий корректен, якорь корректен, mobile CTA и чат корректны.
- `412x915`: overflow нет, фото-сценарий корректен, якорь корректен, mobile CTA и чат корректны.
- `430x932`: overflow нет, фото-сценарий корректен, якорь корректен, mobile CTA и чат корректны.

## Ограничения

- Проверка выполнена локально на production build через `vite preview`. Реальный Vercel CDN и реальные пользовательские RUM-метрики отдельно не измерялись.
- `build:vercel` зависит от локальной авторизации Vercel CLI. Если токен недействителен, команда падает до проверки кода.
